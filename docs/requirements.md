## 1. はじめに (Introduction)

### 1.1 目的 (Purpose)

本書は、個人用「ITニュース集約ポータル」のシステム要件およびソフトウェア要件を定義することを目的とする。複数の IT 系情報メディアに散在する新着記事を一元的に集約・閲覧することで、情報収集に要する手間を削減することを狙いとする。

本書は ISO/IEC/IEEE 29148:2018 が定める要件仕様書の章構成および要件記述の品質特性（明確性・検証可能性・追跡可能性等）を参考に作成する。

### 1.2 対象範囲 (Scope)

#### 1.2.1 対象システムの名称

ITニュース集約ポータル（以下「本システム」）。

#### 1.2.2 本システムが行うこと

- RSS / Atom フィードからの記事メタデータの定期収集
- 収集記事の一覧表示・検索・タグフィルタ・既読管理・あとで読む管理
- 元記事 URL への遷移

#### 1.2.3 本システムが行わないこと

- 記事本文の保存・全文検索
- 外部公開およびマルチユーザ向けの認証・認可
- 記事の AI 要約生成（初期段階では非対象。9 章の将来構想で扱う）
- 大規模スクレイピングや RSS 非提供サイトに対する積極的なクローリング

#### 1.2.4 期待される効果

巡回対象サイト数の増加に対し、利用者の閲覧操作工数を概ね一定に保てるようにする。

## 2. 全体概要 (Overall Description)

### 2.1 システムの位置づけ (System Perspective)

本システムは、外部の RSS / Atom フィード提供サイト（Qiita、Zenn、ITmedia、@IT、はてなブログ等）を上流に持つ、独立した個人用 Web アプリケーションである。外部公開は行わず、ローカル環境での単独動作を前提とする。

```
[外部フィード提供サイト]
        │  RSS / Atom (HTTPS)
        ▼
[記事収集バッチ (cron + rss-parser)]
        │
        ▼
   [SQLite DB]
        │
        ▼
[Next.js Web アプリ (App Router)]
        │
        ▼
   [利用者 (ブラウザ)]
```

### 2.2 システム機能の概要 (System Functions)

- フィード管理（登録 / 一覧 / 削除 / 有効化制御）
- 記事収集（定期実行による取得とメタデータの永続化）
- 記事一覧表示・検索・タグフィルタ
- 既読管理 / あとで読む管理
- 元記事への遷移

### 2.3 利用者の特性 (User Characteristics)

| ステークホルダー | 役割 | 想定リテラシー |
| --- | --- | --- |
| 開発者本人 | 開発・運用・利用すべてを兼ねる単独利用者 | IT エンジニア。RSS、CLI、Next.js 等を理解している前提 |

### 2.4 運用環境 (Operating Environment)

| 区分 | 想定環境 |
| --- | --- |
| 実行端末 | 開発者個人の PC（macOS を想定。Linux / Windows でも動作可能性を残す） |
| ランタイム | Node.js LTS |
| ブラウザ | Chromium 系最新版 |
| データストア | SQLite（ファイルベース） |
| ネットワーク | アウトバウンド HTTPS（フィード取得）が利用可能であること |

### 2.5 設計上の前提と依存 (Assumptions and Dependencies)

- 対象サイトが RSS / Atom フィードを提供し続けること。
- ローカル PC が定期実行（cron 等）を許容する環境であること。
- 認証・外部公開・マルチユーザ対応は初期スコープ外であり、追加要求があった時点で再評価する。

## 4. 機能要件 (Functional Requirements)

各要件は次のフォーマットで記載する。

> **FR-xx (要件名)**
> 概要 / 入力 / 出力 / 主処理 / 関連要件 / 受入基準（検証方法）

### 4.1 RSS フィード管理

#### FR-01 RSS フィードの登録

- **概要**: 利用者は RSS フィードを登録できること。
- **入力**: RSS URL（必須、http(s) スキーム）、フィード名（必須、1〜100 文字）。
- **出力**: 登録結果（成功 / エラーメッセージ）、最新のフィード一覧。
- **主処理**: URL のフォーマット検証 → フィード到達性の簡易確認 → `feeds` テーブルへの永続化。
- **関連**: DR-02
- **受入基準**: 正常な RSS URL を登録すると `feeds` に 1 件追加され、一覧画面に反映される。不正な URL はエラー表示され登録されない。

#### FR-02 RSS フィードの削除 / 無効化

- **概要**: 登録済みフィードを削除、または `is_active = false` で無効化できること。
- **入力**: フィード ID、操作種別（delete / deactivate）。
- **出力**: 操作後のフィード一覧。
- **主処理**: 該当レコードの削除または `is_active` の更新。
- **関連**: DR-02、FR-04（無効化したフィードは収集対象外）
- **受入基準**: 削除後または無効化後、当該フィードからの新規収集が行われないこと。

#### FR-03 RSS フィード一覧の表示

- **概要**: 登録済みフィードを一覧表示できること。
- **出力**: フィード名、RSS URL、有効状態。
- **受入基準**: 登録済みフィードがすべて表示され、状態（有効 / 無効）が判別できる。

### 4.2 記事収集機能

#### FR-04 記事の定期収集

- **概要**: 有効化されているフィードから記事メタデータを定期取得し、`articles` テーブルへ保存できること。
- **取得項目**: タイトル、URL、公開日時、サイト名（フィード名で代用可）、タグ（取得可能な場合のみ）。
- **主処理**: cron（または同等スケジューラ）から収集処理を起動し、各フィードに対し `rss-parser` で取得 → 重複判定（URL ベース） → 新規分のみ INSERT。
- **本文**: 取得しない（CR-02）。
- **関連**: DR-01、NFR-02、OR-01、OR-02
- **受入基準**: 同一 URL の記事が複数回登録されない。新着が無いフィードについては記事数が増加しない。

#### FR-05 取得失敗時のリカバリ

- **概要**: フィード取得・パースに失敗した場合でも、他のフィードの収集を継続できること。
- **主処理**: フィード単位で例外を捕捉し、ログ出力後に次のフィードへ進む。
- **関連**: OR-02
- **受入基準**: 1 フィードが 5xx を返しても、他フィードの記事は問題なく保存される。

### 4.3 記事閲覧機能

#### FR-06 記事一覧の表示

- **概要**: 収集済み記事を一覧表示できること。
- **表示項目**: タイトル、サイト名、投稿日時、タグ、既読状態。
- **既定の並び順**: 公開日時の降順。
- **関連**: NFR-01
- **受入基準**: 既読の記事と未読の記事が視覚的に判別できる。

#### FR-07 記事のキーワード検索

- **概要**: 記事をキーワード検索できること。
- **検索対象**: タイトル、タグ。
- **主処理**: 部分一致検索（大文字小文字を区別しない）。
- **受入基準**: タイトルまたはタグに検索語を含む記事が結果に表示され、含まないものは表示されない。

#### FR-08 タグフィルタ

- **概要**: タグによる絞り込みを行えること（例: AWS、Python、AI、RAG、Snowflake）。
- **入力**: タグ名（複数指定時は OR 条件を初期実装とする）。
- **受入基準**: 指定タグを含む記事のみが一覧に表示される。

#### FR-09 元記事への遷移

- **概要**: 一覧から記事を選択すると、元記事の URL へ新しいタブで遷移できること。
- **主処理**: `<a target="_blank" rel="noopener noreferrer">` 相当のリンクで遷移。
- **受入基準**: クリック時に元記事 URL が正しく開き、本システム側はその記事を既読としてマーク可能であること（FR-10 参照）。

### 4.4 状態管理機能

#### FR-10 既読 / 未読管理

- **概要**: 記事の既読・未読を切り替えられること。元記事遷移時に自動で既読化することを推奨動作とする。
- **入力**: 記事 ID、状態（read / unread）。
- **受入基準**: 状態変更後、一覧の表示が更新される。

#### FR-11 あとで読む管理

- **概要**: 記事を「あとで読む」としてマークし、専用ビューで参照できること。
- **入力**: 記事 ID、状態（save / unsave）。
- **受入基準**: 「あとで読む」一覧で該当記事のみ表示される。

## 7. データ要件 (Data Requirements)

### 7.1 論理データモデル

```
+--------+        +-----------+
| feeds  | 1 ── * | articles  |
+--------+        +-----------+
```

`feeds` と `articles` は実装上はゆるい関連（`articles.source` にフィード名を保持する形）で開始し、将来的な正規化に備える。

### 7.2 articles テーブル (DR-01)

| カラム | 型 (想定) | NOT NULL | 内容 |
| --- | --- | --- | --- |
| id | INTEGER (PK / AUTOINCREMENT) | ✓ | 記事 ID |
| title | TEXT | ✓ | タイトル |
| url | TEXT (UNIQUE) | ✓ | 元記事 URL（重複判定キー） |
| source | TEXT | ✓ | メディア名（フィード名） |
| published_at | DATETIME | ✓ | 公開日時 |
| tags | TEXT (JSON または カンマ区切り) | | タグ |
| is_read | BOOLEAN (DEFAULT 0) | ✓ | 既読フラグ |
| is_saved | BOOLEAN (DEFAULT 0) | ✓ | あとで読むフラグ |
| created_at | DATETIME | ✓ | レコード作成日時（収集日時） |

### 7.3 feeds テーブル (DR-02)

| カラム | 型 (想定) | NOT NULL | 内容 |
| --- | --- | --- | --- |
| id | INTEGER (PK / AUTOINCREMENT) | ✓ | フィード ID |
| name | TEXT | ✓ | フィード名 |
| rss_url | TEXT (UNIQUE) | ✓ | RSS URL |
| is_active | BOOLEAN (DEFAULT 1) | ✓ | 有効フラグ |
| created_at | DATETIME | ✓ | 登録日時 |

### 7.4 データ整合性ルール

- **DR-03**: `articles.url` は一意であり、重複登録を禁止する。
- **DR-04**: `feeds.rss_url` は一意であり、同一 URL の二重登録を禁止する。
- **DR-05**: フィード削除時、関連記事の物理削除は行わない（履歴を保持する）。

## 11. 検証・受入基準 (Verification & Validation)

29148 の「要件は検証可能でなければならない」という品質特性に従い、各要件には受入基準もしくは計測可能な閾値を付している。総括として、本システムは以下を満たした時点で MVP 完了とみなす。

- MVP 範囲の機能要件すべてが、それぞれの受入基準を満たすこと。
- NFR-01 を、ローカル実行環境で実測により満たすこと。
- OR-02 のログが出力され、想定された失敗ケース（フィード 404 / 接続不能 / パース失敗）でアプリ全体がクラッシュしないこと。
